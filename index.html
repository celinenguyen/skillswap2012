Separation of style & content

HTML is meant to focus entirely on meaning—what your content is, what kind of content, how it's structured. CSS is to make it look beautiful. Separating your CSS from your content can be very powerful, since it makes changing the appearance of a website quite effortless—see the CSS Zen Garden for an example.

HTML

Syntax

Here's a totally barebones HTML page that has no content, to give you an idea of what you need to "set up" a webpage. This setup is a bit more extensive, and probably closer to what you'd use in production.

The syntax for HTML is pretty straightforward. It's not a very fussy markup language:

<!-- <element attribute="value"> --> <a href="http://www.scottylabs.org/" title="Organizers of SkillSwap weekend">Scotty Labs</a>

Note some syntactical rules:

  • Put quote marks around your attribute values.
  • If you open a tag, close it. Most tags contain content (for the link, it's the actual link text). ("Empty" elements (e.g. an <img> tag) are the exceptions; they politely close themselves on the way out.
  • Close tags in the right order. They're nested. Close "inside" tags before you close "outside" tas.
  • Comments are indicated with <!-- comment text -->.

Certain characters (symbols that have a special meaning in HTML, e.g. < > &, or non-standard symbols like ∞ ↩ ¾), need to be represented as an HTML character entity. Here's a reference list, and here's a website to look up characters and copy their HTML character codes.

&lt; <!-- represents the < symbol -->

Bread-and-butter elements

See an example of all the tags mentioned below!

  • Document structure: tags like header, section, footer, nav, article, aside are useful in establishing the information structure of your page. They pair quite well with…
  • Headings! Headings go from <h1> to <h6>. Use them to establish a hierarchy (<h1> for the page heading, <h2> for section headings, <h3> for subsections…
  • Paragraphs are your friendly neighborhood <p> tags.
  • Blockquotes, using <blockquote> are for multiline quotes.
  • Inline text tags: strong, em are the most commonly used, but there are a few more to format quotes, citations, deleted and inserted text…
  • Images with the <img src="http://placehold.it/200x200"> tag. Only put content images in your HTML, not ones that are decorative!
  • Links to other websites or different sections of your page.
  • Lists, either unordered with <ul> or ordered with <ol>.
  • Horizontal rules are to separate thematically distinct content within the same section. You can use them to divide sections of content, but that might be a job for CSS to handle (by specifying a border between the sections).

This is not a comprehensive list! But the MDN maintains a list of HTML elements, as well as a list of attributes.

Styling hooks

There are two kinds of elements —block-level and inline elements. Block elements are things like headings, paragraphs, and images—they're on their own line, and what it means for us is that we can position them relatively independently. Inline elements are things like strong or emphasized text—these elements are in line with the other stuff in a paragraph, and their position is dictated by how the other inline elements in the container flow. There's also something weird called inline block elements, which we'll talk about when we hit CSS layouting.

Divs and spans

Think of these as transparent containers that you can apply styles to. They don't have any browser formatting; they're blank slates. Well-placed divs and spans open up a lot of possibility in visuals, but try to not clutter your code with too many!

A div can be used for…a gigantic wrapper or container that boxes in your content and centers it. To indicate chunks of text that are styled differently (for example, an error message). Before the <section> tag was introduced these were used to denote divisions of content.

Spans are inline elements! You can use them to style a few words out of a heading or paragraph differently. I'm using a <span class="amp"> to style all the ampersands on this page. Check it out: &

Classes and IDs

You saw a peek of this above. Classes and IDs are names you can attach to elements if you want to style them!

Multiple elements (and elements of different types) can have the same class. If you're a fine upstanding citizen of the web, only one element on a page can have a particular ID.

CSS

Syntax

CSS is also quite simple:

/* it goes like this: selector { property: value; } */ body { background-color: green; }

Classes and IDs

.class { /* stylin' */ } #id { /* hipster stylin' */ }

Property shorthand

For certain related properties you don't have to declare them one-by-one; they're grouped together into one line. These properties are explained later, but have a go at them now! These are the most commonly used, but here's a list of more CSS shorthand.

/* border shorthand: width style color */ border-width: 1px; border-style: dashed; border-color: green; /* do this instead! */ border: 1px dashed green; /* hex code shorthand: one digit for R, one for G, one for B*/ color: #1188aa; /* do this instead! */ color: #18a; /* DIMENSION SHORTHAND USED FOR: MARGIN, PADDING, BORDER-WIDTH */ /* dimension shorthand: top right bottom left, or TRouBLe */ margin: 30px 50px 20px 50px; /* dimension shorthand: top right&left bottom */ padding: 30px 50px 20px; /* dimension shorthand: top&bottom left&right */ border-width: 10px 5px; /* dimension shorthand: top&bottom&left*right */ margin: 10px;

Selectors

  • Element type: p
  • IDs: #container { … }
  • Classes: .alert
  • Descendant: #container a
  • Direct descendant: section > ul
  • Adjacent: h3 + h4
  • And some special link selectors: a:hover, a:active, a:visited

There are even more fancy selectors (here's a comprehensive list, and here are some thorough articles on more complicated selectors).

Cascading?

CSS style declarations cascade. Browsers use the order and proximity in which styles are defined to determine which one applies to an element! This is helpful for styling the same elements differently in different contexts, for overriding styles, and screwing around with webpages on the client side.

For order, styles declared later (aka styles closer to the end of a list of CSS declarations) override styles declared earlier!

For proximity, note that there are a few ways to include CSS, and this is their descending order of importance dominance:

  1. User-defined styles (there are browser extensions that allow you to restyle pages to your heart's content).
  2. Inline styles (<element style="color: red;">)—note that this breaks the separation of style and content, so this is normally very bad practice.
  3. Page styles, seen as style declarations that you put in a <style type="text/css">/* properties here */</style> element within a page's <head>.
  4. External styles, such as CSS stylesheets that you link from the <head> of a page with the line <link rel="stylesheet" href="css/style.css">.
  5. Browser styles. This is the Times New Roman, kind of ugly styling that the browser applies by default. Note that different browsers can have different default styles…we'll touch on this later.

Specificity

So. We have classes, and IDs, and element types. What if you write styles using different selectors that all end up targeting the same attribute on the same element? The most specific selector wins.

  • IDs are as specific as you can get; there's only one element of a particular ID on the page, so any ID-specific styling takes precedence.
  • Classes (and pseudoclasses) take precedence after IDs.
  • Element selectors are the least targeted and therefore the least important in specificity calculations.
  • More generic styles will be overridden for more targeted styles. A style for a link only inside unordered lists is "stronger" than a style for links in general.
  • Don't make selectors too specific (like using ul.recipeingredients to select an unordered list with the class recipeingredients if that class is only used for unordered lists!).

You can learn how specificity is calculated, but common sense is probably enough, and this is only a major issue if you have overly complex and specific selectors, or you're working on a very large stylesheet.

Messy hacks (!important)

In quite a few cases you can subvert the usual cascade/specificity by using the keyword !important to force a particular style to be expressed. It's kind of hackish I'd recommend using it only in debugging.

element { color: red !important; }

Positioning elements

  • Normal flow is where block elements will stack vertically on a page (staying within their containing blocks), and inline elements will flow horizontally left to right.
  • Absolutely positioned elements are taken out of the normal flow and put at a precise pixel or percentage location relative to some container.

Color

  • Hex codes like #ff0000 (pure red), or #666666 (dark grey, which can be shortened to #666). The first pair of digits are the amount of red , the second pair blue, and the third pair the amount of green in the color.
  • RGB values, which are specified as rgb(0, 255, 30) (again: red value, green value, blue value, from 0 to 255).
  • RGBA values, which also specify an alpha transparency on a scale of 0 (transparent) to 1 (opaque). rgba(0, 255, 30, 0.5) is 50% transparent.
  • Named colors, of which there are 147. Standard colors like red, pretentious colors like palegoldenrod and obscure colors like bisque are included.

Here's a color palette tool that gives you hex and RGB values, and here's a useful color blender.

Units

  • Pixels, which are quite straightforward: 200px.
  • Percentages of an element's container (or parent element), e.g. 60%.
  • Ems, which are relative units. 1em is equal to the font-size of an element's container. Here's a somewhat old-school article on using ems for elastic layouts.
  • Rems, which are ems indexed from the font-size of the <body> element. Here's an article on when rems are useful.

Font stacks

Since different OSes and computers will have a different set of fonts installed, specify a list of fonts (a font stack) for a browser to try to display text with. The list starts off with preferred, possibly-unavailable typefaces and proceeds to more basic and widely available typefaces.

A quick note on default styling and designing for different browsers

CSS resets

Different browsers have different ways of interpreting your markup

There are essentially four rendering engines: Webkit (Chrome and Safari both use this, so they'll render webpages almost identically), the rendering engine behind Firefox, the rendering engine behind Opera, and the rendering engine behind Internet Explorer. Internet Explorer in particular has some very annoying quirks, especially in the older versions. Read this to get a look into what some rendering differences are and how to deal with them.

If you want to get really fancy, you'll also have to consider how mobile users view a website compared to desktop users.

I want variables in CSS!

If you've ever programmed before, or if you start writing a ton of CSS, things will start to bug you. Like not being able to give a hex code a name if you're using a specific shade of blue 34 times in your code.

This and other problems can be solved with CSS preprocessors, which allows you to write in a more full-featured, modified syntax that eventually compiles back to CSS. The most-used preprocessors are LESS and SCSS.
